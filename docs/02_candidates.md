# 후보 비교

[설계 문서](01_design.html)에서 정의한 흐름을 기준으로, 통신사 연동 방식 후보 3가지를
비교합니다.

## 비교표

| 항목 | A. 직접 API 연동 | B. SDK 임베드 | C. 딥링크 리다이렉트 |
| --- | --- | --- | --- |
| 개발 난이도 | 높음 | 중간 | 낮음 |
| 사용자 경험 | 매끄러움 | 매끄러움 | 이탈 발생 가능 |
| 통신사 측 작업량 | 많음 | 중간 | 적음 |
| 보안 검증 범위 | 넓음 | 중간 | 좁음 |
| 유지보수 비용 | 높음 | 중간 | 낮음 |

## 세부 검토

### A. 직접 API 연동

통신사 서버가 OGQ API를 직접 호출합니다.

```bash
curl -X POST https://api.ogq.com/v1/partner/subscribe \
  -H "Authorization: Bearer <partnerToken>" \
  -d '{"userId": "u123", "planCode": "premium"}'
```

- 장점: 사용자 경험이 가장 매끄럽습니다.
- 단점: 통신사 측 개발 리소스가 많이 필요합니다.

### B. SDK 임베드

OGQ가 제공하는 SDK를 통신사 앱에 임베드합니다.

- 장점: 개발 난이도와 통신사 부담의 균형이 좋습니다.
- 단점: SDK 버전 관리 이슈가 발생할 수 있습니다.

> [!TIP]
> 세 후보 중 가중 점수가 가장 높아 실제로 채택된 방식입니다. 자세한 근거는 [결정 게이트](03_decision_gate.html) 문서를 참고하세요.

### C. 딥링크 리다이렉트

통신사 앱에서 OGQ 페이지로 딥링크 이동 후 처리합니다.

- 장점: 구현이 가장 빠릅니다.
- 단점: 앱 전환으로 인한 사용자 이탈 우려가 있습니다.
